# CI Pipeline 繳交報告（b11705027）

## 1. CI Pipeline 說明

### 1.1 Workflow 設定檔

檔案路徑：`.github/workflows/ci_b11705027.yaml`

```yaml
name: CI b11705027

on:
  push:

permissions:
  contents: read
  checks: write

jobs:
  ci:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v5

      - name: Setup Node.js
        uses: actions/setup-node@v5
        with:
          node-version: '22'
          cache: npm

      - name: Install dependencies
        run: npm ci

      - name: TypeScript typecheck
        run: npm run typecheck

      - name: Prettier check
        run: npm run format:check

      - name: Run tests
        id: tests
        run: |
          mkdir -p reports
          npm test -- --reporter=default --reporter=junit --outputFile.junit=reports/vitest-junit.xml

      - name: Publish test report
        if: steps.tests.outcome == 'success' || steps.tests.outcome == 'failure'
        uses: dorny/test-reporter@v2
        with:
          name: Vitest Tests
          path: reports/vitest-junit.xml
          reporter: java-junit
          fail-on-error: true
```

### 1.2 Pipeline 設計說明

| 項目                    | 實作方式                                                                                                                                                                                                                    |
| ----------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **觸發條件**            | `on: push`：每次 push 到 repository 的任何分支都會自動執行。                                                                                                                                                                |
| **執行環境**            | `ubuntu-latest` runner，Node.js 22（符合 `package.json` 的 `engines`）。                                                                                                                                                    |
| **相依安裝**            | `npm ci` 依 `package-lock.json` 安裝，確保 CI 與本機一致。                                                                                                                                                                  |
| **TypeScript 型別檢查** | `npm run typecheck`（`tsc --noEmit`），型別錯誤時步驟 exit code 非 0，整個 job 失敗。                                                                                                                                       |
| **Prettier 格式檢查**   | `npm run format:check`（`prettier --check .`），格式不符時 job 失敗。                                                                                                                                                       |
| **單元測試**            | `npm test`（Vitest），並以 JUnit XML 輸出到 `reports/vitest-junit.xml`。                                                                                                                                                    |
| **失敗處理**            | 任一步驟失敗即中止後續步驟（預設行為），workflow run 狀態為 **failure**。                                                                                                                                                   |
| **測試結果呈現**        | 使用 [dorny/test-reporter](https://github.com/marketplace/actions/test-reporter) 將 JUnit 報告發佈為 GitHub Check，在 Actions 執行頁的 **Tests** 分頁可檢視通過/失敗的測試案例。僅在測試步驟有執行時才發佈報告。 |

### 1.3 工具與策略

- **GitHub Actions**：雲端 CI，無需自建 runner。
- **actions/checkout、actions/setup-node**：官方維護的標準步驟，並啟用 npm cache 加快安裝。
- **Vitest JUnit reporter**：與 Lab-02 snippet 相同，產生 `vitest-junit.xml` 供第三方 action 解析。
- **dorny/test-reporter**：將 JUnit 轉成 GitHub Checks API 的測試摘要，滿足「在 Actions 結果頁顯示測試結果」的要求。
- **單一 job、循序步驟**：邏輯清楚，任一檢查失敗即讓整條 pipeline 失敗，符合作業「失敗處理」要求。

## 3. 失敗案例說明

以下提供三種可故意製造失敗的方式，擇一實作並截圖即可（作業要求至少一種失敗案例）。

### 3.1 案例 A：TypeScript 型別錯誤

**作法**：在 `src/app.ts` 暫時加入：

```typescript
const broken: number = 'not-a-number';
```

**預期結果**：`TypeScript typecheck` 步驟失敗，workflow 顯示 failure。

**原因**：將字串指派給 `number` 型別變數，違反 TypeScript 型別規則。

**修復**：刪除該行或改為正確型別，重新 commit 並 push。

---

### 3.2 案例 B：Prettier 格式錯誤

**作法**：將某段程式碼改成不符合 `.prettierrc` 的格式，例如故意去掉分號、錯誤縮排，且不要執行 `npm run format`。

**預期結果**：`Prettier check` 步驟失敗。

**原因**：`prettier --check` 比對專案格式規範，發現差異即 exit 1。

**修復**：執行 `npm run format` 自動修正，或手動調整後再 push。

---

### 3.3 案例 C：測試失敗

**作法**：在 `test/app.test.ts` 暫時修改斷言，例如：

```typescript
expect(response.statusCode).toBe(500);
```

**預期結果**：`Run tests` 步驟失敗；若已產生 JUnit 檔，`Publish test report` 仍會執行並在 **Tests** 顯示失敗案例。

**原因**：實際回應為 200，與預期 500 不符。

**修復**：還原正確的 `expect(response.statusCode).toBe(200)`，重新 push。
